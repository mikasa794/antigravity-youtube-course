## 核心观点
*   **AI 作为“结对工作”伙伴**：Teresa 将 Claude Code 视为一个可以“结对编程”、“结对写作”、“结对思考”的全能伙伴，而不仅仅是代码助手。
*   **上下文分层管理是关键**：通过建立三层上下文系统（全局偏好、项目特定规则、业务/个人档案），她让 Claude 在不同任务中都能精准调用所需信息，避免“上下文污染”。
*   **从“解释”到“构建记忆”**：核心工作流是，每当需要向 Claude 重复解释某件事时，就停下来，与 Claude 协作创建一个可复用的“上下文文件”，实现知识的沉淀和自动化。
*   **过程重于结果，人保持主导**：在写作等高价值任务中，她让 Claude 担任研究员、编辑和思维碰撞伙伴，但坚持自己完成核心创作，以保持个人风格和深度思考。
*   **迭代构建，而非一蹴而就**：整个复杂系统（任务管理、研究、写作）是随着日常需求，通过“这个任务 Claude 能怎么帮忙？”的提问，逐步、迭代地搭建起来的。

## 关键洞察
**“上下文工程”是新的“提示工程”**：Teresa 的工作流精髓不在于编写复杂的单次提示，而在于构建一个**可持续、可扩展的上下文生态系统**。她将关于自己、业务和偏好的信息模块化，存储为独立的 Markdown 文件（如“业务概况”、“目标受众”、“写作风格指南”）。Claude 通过一个“索引文件”知道这些文件的存在和位置，并能根据当前任务动态加载相关模块。这极大地降低了每次交互的认知负荷，让 Claude 的表现从“偶尔惊艳”变得“持续可靠”。这种思路将 AI 从“一次性工具”转变为拥有“组织记忆”的长期协作伙伴。

**“生活在技术前沿”的产品思维**：Teresa 认为，产品经理如果要构建 AI 产品，就必须亲自深度使用并“生活在 AI 能力的前沿”。她通过每日使用 Claude Code 遇到的挑战（如上下文窗口管理、权限设置），获得了对 LLM 局限性和潜力的第一手深刻理解。这种经验无法从阅读报告或使用封装好的 SaaS 工具中获得。她将使用 Claude Code 的磕绊过程，视为构建更健壮 AI 产品的必要训练，体现了“以用户（也是建造者）身份深入场景”的务实产品哲学。

## 详细摘要
本期节目深入探访了知名产品教练、《持续发现习惯》作者 Teresa Torres 如何将 Anthropic 的 Claude Code 深度整合到她的工作与生活中，将其从一个代码助手转变为管理任务、驱动研究、辅助写作的全能“数字员工”。

### 工作流全景：Obsidian + Claude Code 的双核系统
Teresa 的核心工作环境由左侧的 Obsidian（基于 Markdown 的笔记工具）和右侧的两个终端窗口（运行 Claude Code）构成。她为不同工作领域创建了独立的“项目文件夹”（如 `tasks`, `writing`），每个文件夹都有专属的 `claude.md` 文件，定义了在该上下文中与 Claude 协作的规则。

### 自动化任务与日程管理
*   **每日待办清单**：每天早晨，她在 `tasks` 文件夹中键入 `today`。Claude 会执行一个 Python 脚本：1) 检查 Trello 看板是否有团队新任务；2) 扫描本地 `tasks` 文件夹，找出所有到期和逾期的任务（每个任务是一个 Markdown 文件）；3) 生成一份结构清晰的 `today.md` 待办清单，并显示在 Obsidian 中。
*   **快速任务捕获**：在工作中，任何新想法或任务，她都可以在终端里直接输入 `new task [描述] due [日期]` 或 `new idea [描述]`，Claude 会瞬间创建对应的 Markdown 文件并放入相应文件夹，流程极简高效。
*   **每日研究简报**：`today` 命令还会触发一个研究系统，自动搜索 arXiv 和 Google Scholar，抓取与她工作相关的最新学术论文，并生成摘要链接加入待办清单。

### 与 Claude “结对写作”的深度协作
Teresa 以共同撰写一篇关于 Claude Code “计划模式”和“自动接受模式”的博客文章为例，展示了她的写作流程：
1.  **启动与规划**：在 `writing` 文件夹中启动 Claude，给出博客主题和初步想法。Claude 首先会读取她的“写作风格指南”，然后进入“计划模式”共同构思大纲。
2.  **研究与碰撞**：她会要求 Claude 进行竞争性分析（“有没有人写过类似主题？”）、关键词研究、甚至核实某个观点的学术依据。Claude 扮演研究员角色。
3.  **撰写与反馈**：Teresa 亲自撰写每个部分以保持个人风格和思考深度。写完一段后，立即让 Claude 审阅，获取关于逻辑、清晰度、技术准确性的反馈，并修正拼写错误。这是一个紧密的、段落级的“撰写-反馈”循环。
4.  **保持动量**：Claude 会在每个环节后主动询问“准备好进入下一阶段了吗？”，这种推动力帮助她保持专注，极大提升了写作效率。她提到，一篇 9000 字、信息密度极高的深度指南，可以在一天半内完成，这在以前是不可想象的。

### 核心心法：三层上下文与记忆管理
Teresa 成功的关键在于卓越的“上下文工程”：
1.  **全局偏好 (`~/.claudemd`)**：包含适用于所有场景的顶层规则，如“做事前必须先计划”，非常精简。
2.  **项目特定规则 (如 `writing/claude.md`)**：定义在特定领域（如写作）如何协作，例如“每次会话开始前先读我的风格指南”。
3.  **业务/个人上下文文件**：这是一个由众多小文件组成的“记忆库”，存放在 `llm_context` 文件夹中，涵盖公司概况、产品描述、团队信息、差异化优势等。关键技巧是：
    *   **动态加载**：通过一个“索引文件”告诉 Claude 这些文件的存在，Claude 只在需要时读取相关文件，避免上下文窗口被无关信息挤占。
    *   **让 Claude 维护**：她从不手动创建或更新这些文件。每当与 Claude 协作后，她会问：“你从这次对话中学到了什么关于我/业务的新东西？我们应该把它加到哪个上下文文件里？” 或者直接说“采访我关于我的业务”，让 Claude 通过提问来生成或更新档案。
    *   **核心原则**：每当发现自己要向 Claude 重复解释某件事时，就停下来，转而与 Claude 协作将这部分信息固化到一个上下文文件中。

### 给新手的建议与哲学
*   **从小处开始**：不必一次性构建复杂系统。只需设立 `work` 和 `personal` 两个基础文件夹，然后在日常工作中，坚持问一个问题：“**这个任务 Claude 能怎么帮忙？**”
*   **区分“自动化”与“增强”**：对于热爱的任务（如写作），思考如何用 AI **增强**（研究、编辑、提供灵感）；对于繁琐的任务，思考如何**自动化**。
*   **像管理员工一样管理 AI**：将 Claude 视为团队新成员。良好的委派需要清晰的操作流程和背景信息。你在人类助理身上积累的“委托肌肉”，同样适用于 AI。
*   **为前沿付费**：Teresa 认为，产品人若想打造优秀的 AI 产品，必须亲身“生活在技术前沿”，体验并解决使用中的痛点（如上下文窗口限制），这能带来无可替代的深度认知。Claude Code 正是这样一个能让你触及前沿的工具。
